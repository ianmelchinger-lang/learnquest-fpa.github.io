Driver-based planning

Find the true drivers in a real payments book

A static budget is a list of numbers: no formula connects them, so a "what if" question means rebuilding by hand. A driver-based budget is built in layers instead \u2014 a small set of operational inputs you can set, and a chain of formulas that calculates everything else from them. The hard part is knowing which is which. Below is a real payments business's numbers, drawn straight from its books. Your job is to sort them into what you can set and what the model calculates for you.

FP&A stands for financial planning and analysis. A budget organized by accounting category \u2014 a fixed number for "revenue," another for "cost" \u2014 cannot answer a scenario question, because nothing in it says how one number moves another. A budget organized around drivers can, because the drivers are the causes and the rest of the model is built to calculate the effects. A few terms before you start.

Driver
An operational number you set as an assumption \u2014 change it, and everything downstream recalculates.
Calculated output
A number the model produces from the drivers above it. You read it; you don't set it directly.
Take rate
The share of each dollar of payment volume the company keeps as revenue.
Cost rate
A cost expressed as a percent of revenue, so it scales automatically as revenue moves.
Driver-based model
A model built in three layers: drivers you set, formulas that calculate from them, and the results those formulas produce.
